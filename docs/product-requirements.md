# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**YouTube 新着要約通知エージェント** - 監視チャンネルの新着動画を毎晩要約してメール通知

### プロダクトコンセプト
- **無認証・低コスト運用**: YouTube の無認証 RSS で新着を検出し、YouTube Data API / OAuth を使わない。鍵は Gemini API キー1つと Gmail アプリパスワードのみ。
- **決定的なパイプライン**: 毎回 LLM に段取りを考えさせるエージェント型ではなく、手順を固定した決定的スクリプト（RSS 検出 → Gemini 要約 → HTML メール送信）。Claude Code はあくまで開発ツールとして使う。
- **取りこぼしゼロの通知**: 既読 ID 記録方式で二重通知と取りこぼしを防止。要約できない動画もタイトル・リンク付きで必ず通知に含める。

### プロダクトビジョン
YouTube を毎日巡回せずとも、監視対象チャンネルの新着動画の内容を "観なくても分かる" レベルで把握できる状態を提供する。
毎晩22時に自動で新着を検出・要約し、1通のメールにまとめて届けることで、情報収集にかかる時間と見逃しをゼロに近づける。
運用は GitHub Actions 上で完結し、サーバー管理・課金の心配なく個人が持続的に利用できる。

### 目的
- 監視チャンネルの新着動画を毎日確実に検出し、見逃しをなくす。
- 各動画を日本語で構造化要約し、動画を再生せずに内容を判断できるようにする。
- 通知を1日1通のメールに集約し、通知疲れを防ぐ。
- API キー・OAuth・サーバーを増やさず、無料枠中心で持続運用する。

## ターゲットユーザー

### プライマリーペルソナ: 本ツールの所有者（個人利用者、`you@example.com`）
- 特定の YouTube チャンネル（技術・情報系など）を継続的に追いたいが、毎日巡回する時間がない。
- 新着を見逃したくない一方で、通知が頻繁に来るのは避けたい。
- サーバー運用や課金管理の手間はかけたくなく、無料枠・無認証で完結する仕組みを好む。
- GitHub / Python / CLI の基本操作はできる。監視チャンネルの追加は自分でコマンドを叩ける。
- 典型的なワークフロー: 日中は別作業に集中し、夜にメールを1通確認して「観るべき動画」を判断する。

### 想定利用規模
- 利用者: 1名（自分専用）。マルチユーザー・共有は想定しない。
- 監視チャンネル数: 数〜数十チャンネル。
- 1日あたり新着: 0〜十数本程度。

## 成功指標(KPI)

### プライマリーKPI
- **取りこぼしゼロ**: RSS に現れた新着動画のうち、通知に含まれる割合 100%（要約不能なものも代替表示で必ず含める）。
- **二重通知ゼロ**: 一度通知した動画が再通知される件数 0 件（`seen.json` による既読管理で担保）。
- **定時実行成功率**: scheduled workflow の成功率 95% 以上（多少の遅延は許容、失敗時は翌日リカバリ）。

### セカンダリーKPI
- **要約成功率**: 新着動画のうち Gemini 要約が生成できた割合 80% 以上（メンバー限定・非公開・年齢制限等を除く）。
- **通知の集約**: 新着がある日は1日1通に集約（複数通送信 0 件）。新着0本の日は送信 0 通。
- **運用コスト**: 月額の追加課金 0 円（Gemini 無料枠・Gmail 送信制限内で運用）。

### 測定方法
- **定時実行成功率**: GitHub Actions の workflow 実行履歴（成功/失敗）で集計。失敗時は GitHub の通知メールで検知。
- **取りこぼしゼロ / 二重通知ゼロ**: 実行ログの処理件数と `seen.json` の Git 履歴（コミット差分）を受信メールと突合して確認。
- **要約成功率**: 実行ログに要約成功/失敗（`summary_ok`）の件数を出力し、Actions ログで集計。
- **運用コスト**: Gemini API / GitHub の利用状況ダッシュボードで無料枠内であることを月次確認。

## 機能要件

### コア機能(MVP)

#### 1. 監視チャンネル管理

**ユーザーストーリー**:
利用者として、追いたいチャンネルを見逃さず監視するために、@handle や URL を渡すだけで監視対象に登録できる仕組みが欲しい。

**受け入れ条件**:
- [ ] ヘルパースクリプトに @handle または チャンネル URL を渡すと、一度だけ channel_id に解決して設定ファイルへ追記する。
- [ ] 監視チャンネルは手動リスト管理とし、登録チャンネルの自動追従は行わない。
- [ ] 実行時（新着検出時）は解決済みの channel_id のみを使用し、handle 再解決を行わない。
- [ ] 同一チャンネルを重複登録しようとした場合は追記せず、その旨を表示する。
- [ ] channel_id に解決できない入力（不正な @handle / URL）はエラーメッセージを表示し、設定ファイルを変更しない。

**優先度**: P0(必須)

#### 2. 新着動画の検出

**ユーザーストーリー**:
利用者として、認証情報を管理せずに新着を取得するために、無認証 RSS による新着検出が欲しい。

**受け入れ条件**:
- [ ] 各チャンネルの無認証 RSS（`https://www.youtube.com/feeds/videos.xml?channel_id=…`）から動画一覧を取得する。
- [ ] API キー・OAuth を一切使用しない。
- [ ] 一部チャンネルの RSS 取得に失敗しても、他チャンネルの処理を継続する（部分失敗を許容）。

**優先度**: P0(必須)

#### 3. 新着判定（既読 ID 記録方式）

**ユーザーストーリー**:
利用者として、取りこぼしと二重通知を防ぐために、通知済み動画を記録して差分だけを新着として扱う仕組みが欲しい。

**受け入れ条件**:
- [ ] `seen.json` に通知済み動画 ID を保存する。
- [ ] RSS に存在し `seen.json` に無い動画のみを新着と判定する。
- [ ] 通知送信後、新着動画 ID を `seen.json` に追記し、状態をリポジトリにコミットして書き戻す。
- [ ] 送信失敗時は `seen.json` を更新せず、翌日以降に再通知できる状態を保つ。

**優先度**: P0(必須)

#### 4. 動画要約（Gemini）

**ユーザーストーリー**:
利用者として、動画を再生せずに内容を把握するために、YouTube URL を渡すだけで日本語の詳しい要約が得られる仕組みが欲しい。

**受け入れ条件**:
- [ ] Gemini API に YouTube URL を直接渡して要約を生成する（字幕スクレイピング不要、Claude API は不使用）。
- [ ] 要約は日本語で、小見出し付きの構造化された "観なくても分かる" 粒度とする。
- [ ] 要約不能な動画（メンバー限定 / 非公開 / 年齢制限 / 長すぎ / API エラー）は「⚠️ 要約できませんでした」と明記し、RSS の概要欄を代替表示する。
- [ ] 要約不能でもタイトル・チャンネル名・リンクは必ず通知に含める（取りこぼしゼロ）。
- [ ] 長尺動画は精度低下・処理時間増を許容する。

**優先度**: P0(必須)

#### 5. メール通知（Gmail SMTP）

**ユーザーストーリー**:
利用者として、新着を1箇所でまとめて確認するために、その日の新着を1通の HTML メールで受け取りたい。

**受け入れ条件**:
- [ ] Gmail SMTP（アプリパスワード）で `you@example.com` から同アドレスへ HTML メールを送信する。
- [ ] 各動画にタイトル・チャンネル名・公開時刻・YouTube リンク・要約（または代替表示）を付与する。
- [ ] その日の新着を1通に集約して送信する。
- [ ] 新着が0本の日は送信しない。

**優先度**: P0(必須)

#### 6. 定時自動実行（GitHub Actions）

**ユーザーストーリー**:
利用者として、手動実行なしで毎晩まとめを受け取るために、定時で自動実行される仕組みが欲しい。

**受け入れ条件**:
- [ ] GitHub Actions の scheduled workflow で毎晩 22:00 JST（cron `0 13 * * *`、UTC 13:00）に実行する。
- [ ] 多少の実行遅延を許容する。
- [ ] Gemini API キー / Gmail アドレス / Gmail アプリパスワードを GitHub Secrets から参照する。
- [ ] 手動トリガー（workflow_dispatch）でも実行できる。

**優先度**: P0(必須)

### CLIインターフェース

```bash
# 監視チャンネルの追加（@handle / URL を channel_id に解決して設定ファイルへ追記）
python -m app.add_channel "@handlename"
python -m app.add_channel "https://www.youtube.com/@handlename"

# 新着検出 → 要約 → メール送信（GitHub Actions からも同じエントリを実行）
python -m app.run
```

### 将来的な機能(Post-MVP)

#### 要約プロンプト / モデルのチューニング
要約の粒度や口調を設定で切り替え、動画カテゴリごとに最適化する。

**優先度**: P2(できれば)

#### 送信・要約の上限運用対応
Gmail の1日あたり送信制限や Gemini 無料枠に触れた場合の分割送信・スロットリング対応。

**優先度**: P2(できれば)

## 非機能要件

### パフォーマンス
- 1日の処理は GitHub Actions の1ジョブ（デフォルトのタイムアウト内、目安6時間以内）で完了する。実運用では新着数十本規模で数分〜十数分を想定。
- 遅延は許容。定時から多少ずれても、その日の新着が漏れなく処理されればよい。

### ユーザビリティ
- 監視チャンネルの追加が1コマンドで完了する。
- メールを開いた瞬間に「観るべき動画」を判断できる構造（タイトル・チャンネル名・公開時刻・要約が一目で分かる HTML レイアウト）。

### 信頼性
- 取りこぼしゼロ・二重通知ゼロ（`seen.json` と、送信成功後にのみ状態を書き戻す設計で担保）。
- 一部チャンネルの RSS 取得失敗・一部動画の要約失敗があっても、処理全体は継続し、成功分は通知する（部分失敗の許容）。
- 送信失敗時は状態を更新せず、翌日リカバリ可能とする。

### セキュリティ
- Gemini API キー・Gmail アドレス・Gmail アプリパスワードは GitHub Secrets で管理し、リポジトリにコミットしない。
- ログに機密情報（アプリパスワード・API キー）を出力しない。

### スケーラビリティ
- 監視チャンネル数十規模までを対象とする。大規模マルチユーザー対応はスコープ外。

## スコープ外

明示的にスコープ外とする項目:
- YouTube の登録チャンネル一覧との自動同期（監視は手動リスト管理）。
- YouTube Data API / OAuth を用いた取得。
- Claude API による要約（要約は Gemini のみ）。
- マルチユーザー・複数宛先への配信、共有機能。
- Web UI / モバイルアプリ（通知はメールのみ）。
- 字幕スクレイピングや音声書き起こしの自前実装。
- リアルタイム / 即時通知（1日1回の定時バッチのみ）。
