# 技術仕様書 (Architecture Design Document)

本書は `docs/product-requirements.md` / `docs/functional-design.md` を技術的に実現するための、技術選定・システム構造・運用制約を定義する。
中心方針は **決定的パイプライン**（LLM に段取りを考えさせない）、**無認証・無サーバー・無料枠中心**、**状態は Git コミットで永続化** の3点。

## テクノロジースタック

### 言語・ランタイム

| 技術 | バージョン |
|------|-----------|
| Python | 3.11 以上（GitHub Actions `ubuntu-latest` 標準） |
| pip | 同梱版 |

### フレームワーク・ライブラリ

| 技術 | バージョン | 用途 | 選定理由 |
|------|-----------|------|----------|
| httpx | ^0.27 | 無認証 RSS / チャンネルページの HTTP 取得 | タイムアウト・リトライ制御が容易。requests でも可 |
| feedparser | ^6.0 | RSS(Atom) のパース | YouTube フィードの解析に実績があり堅牢 |
| google-genai | 最新安定版 | Gemini API へ YouTube URL を渡して要約 | 公式 SDK。URL 直渡し要約に対応。Claude 非使用の方針に合致 |
| （標準）smtplib / email | 標準 | Gmail SMTP による HTML メール送信 | 追加依存不要。アプリパスワードで送信可能 |
| （標準）json / pathlib / logging | 標準 | 設定・状態の入出力、ログ | 依存を増やさない |

> Gemini の**具体モデル**（最新の要約向けモデル）は実装着手時に確定する。SDK バージョンとモデル ID は密結合するため、選定時に `requirements.txt` へ固定する。

### 開発ツール

| 技術 | バージョン | 用途 | 選定理由 |
|------|-----------|------|----------|
| pytest | ^8.0 | ユニット/統合テスト | Python 標準的なテストランナー |
| ruff | 最新 | Lint / フォーマット | 高速・設定が単純 |
| GitHub Actions | - | scheduled 実行・CI | サーバー不要、cron と Secrets を提供 |

## アーキテクチャパターン

### レイヤードアーキテクチャ（パイプライン指向）

Web/UI レイヤーは存在しない。エントリポイント（CLI/Actions）→ オーケストレーション → I/O アダプタ、の3層で構成する。

```
┌─────────────────────────────────────┐
│  エントリレイヤー                     │ ← app.run / app.add_channel（引数受付・ロギング）
├─────────────────────────────────────┤
│  オーケストレーションレイヤー          │ ← Pipeline（固定順の結線・新着判定・状態確定）
├─────────────────────────────────────┤
│  アダプタレイヤー                     │ ← RssFetcher / Summarizer / MailSender / ConfigStore / ChannelResolver
└─────────────────────────────────────┘
        ↕ 外部: YouTube RSS / Gemini API / Gmail SMTP / JSON ファイル
```

#### エントリレイヤー
- **責務**: コマンド引数の受付、環境変数（Secrets）読み込み、ログ初期化、終了コード管理。
- **許可**: オーケストレーションレイヤーの呼び出し。
- **禁止**: 外部 I/O の直接実行、ビジネスロジックの実装。

#### オーケストレーションレイヤー（Pipeline）
- **責務**: 「RSS 取得 → 新着判定 → 要約 → HTML 生成 → 送信 → 状態書き戻し」の固定順制御。送信成功後のみ状態を確定するトランザクション境界を持つ。
- **許可**: アダプタレイヤーの呼び出し。
- **禁止**: HTTP/SMTP/SDK の直接呼び出し（必ずアダプタ経由）。

#### アダプタレイヤー
- **責務**: 外部システム（RSS/Gemini/Gmail）とファイル I/O をカプセル化。失敗の局所化（チャンネル/動画単位の握りつぶし）。
- **許可**: 外部ネットワーク・ファイルシステムへのアクセス。
- **禁止**: パイプライン全体の制御フロー判断。

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| 監視チャンネル一覧 | リポジトリ内ファイル `config/channels.json` | JSON | 機密でなく、変更履歴を Git で追える |
| 既読動画 ID | リポジトリ内ファイル `state/seen.json` | JSON | DB 不要。取りこぼし・二重通知防止をシンプルに担保 |
| 機密情報 | GitHub Secrets → 環境変数 | - | リポジトリに残さない |

### 状態書き戻し（Git コミット）

- `app.run` は **メール送信成功後にのみ** `state/seen.json` を更新する。
- 更新後、workflow 内で `git add / commit / push` により状態をリポジトリへ書き戻す。
- コミットは自動化用のマシンコミット（例: `chore: update seen.json [skip ci]`）とし、`[skip ci]` で無限ループを防ぐ。
- 送信前に異常終了した場合は seen.json を変更しないため、翌実行で同一新着が再処理される（取りこぼし防止）。

### バックアップ戦略

- **バックアップ**: Git 履歴自体が世代管理となる（seen.json の各コミットが復元ポイント）。
- **復元方法**: 破損時は直前コミットへ `git checkout` / `git revert` で戻す。
- **肥大化対策**: seen_ids は上限件数（例: 直近数千件）で古いものを切り詰め可能。

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| 1回の日次実行（新着数〜十数本） | 数分〜十数分 | GitHub Actions `ubuntu-latest` |
| ジョブ全体 | Actions のジョブタイムアウト内（目安 6 時間、実際は大幅に下回る） | 同上 |
| add_channel（1件解決） | 数秒 | ローカル/Actions |

遅延は許容。定時（22:00 JST）から多少ずれても、その日の新着が漏れなく処理されればよい。

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | Actions ランナー既定（7GB 目安）内 | 動画メタデータと要約テキストのみ保持。大容量データを扱わない |
| 実行時間 | 無料枠の月間分数を意識（1日1回・短時間） | 課金 0 円運用 |

## セキュリティアーキテクチャ

### データ保護

- **機密情報管理**: `GEMINI_API_KEY` / `GMAIL_ADDRESS` / `GMAIL_APP_PASSWORD` を GitHub Secrets で管理し、実行時に環境変数として注入。ファイル・ログに書き出さない。
- **アクセス制御**: リポジトリを private 推奨。Secrets はリポジトリ/環境スコープで限定。
- **暗号化**: 通信は HTTPS（RSS/Gemini）と TLS（SMTP over SSL, 465 もしくは STARTTLS 587）。

### 入力検証

- **add_channel**: @handle / URL 形式を検証し、解決できない入力はエラーで停止（設定ファイルを汚さない）。
- **RSS パース**: 欠損フィールドを許容し、必須（video_id/title/url）が欠ける項目はスキップ。
- **エラーハンドリング**: 例外メッセージに API キー・パスワードが混入しないようログを選別。

## スケーラビリティ設計

### データ増加への対応

- **想定規模**: 監視チャンネル数十、1日新着 0〜十数本、seen_ids 数千〜（切り詰めで抑制）。
- **劣化対策**: 要約は動画単位で独立。件数増でも線形増にとどめ、過剰な並列化は行わない（レート制限回避）。
- **アーカイブ**: seen_ids の上限超過分を古い順に削除。

### 機能拡張性

- **設定のカスタマイズ**: 監視リストは `channels.json` で完結。要約プロンプト/モデルは設定または定数で差し替え可能な構造にする（Post-MVP でのチューニング余地）。
- **拡張方法**: アダプタ差し替え（例: 送信先を SMTP 以外へ）で機能追加できる疎結合構成を維持。

## 運用監視・KPI 測定

PRD の KPI（`docs/product-requirements.md` 参照）を、追加インフラなしで以下の手段で測定する。

| KPI | 測定手段 |
|-----|---------|
| 定時実行成功率 | GitHub Actions の workflow 実行履歴（成功/失敗）。失敗時は GitHub 通知メールで検知 |
| 取りこぼしゼロ / 二重通知ゼロ | 実行ログの処理件数 + `seen.json` の Git コミット履歴を受信メールと突合 |
| 要約成功率 | 実行ログに `summary_ok` の成功/失敗件数を毎回出力し、Actions ログで集計 |
| 運用コスト | Gemini API / GitHub の利用状況ダッシュボードを月次確認 |

実装要件: `app.run` は実行末尾に「処理チャンネル数 / 検出新着数 / 要約成功・失敗数 / 送信有無」のサマリー行をログ出力する。

## テスト戦略

### ユニットテスト
- **フレームワーク**: pytest
- **対象**: 新着判定ロジック、ConfigStore の重複拒否/読み書き、MailBuilder の HTML 生成（要約成功/失敗）、ChannelResolver のパース。
- **カバレッジ目標**: コアロジック（判定・状態・HTML 生成）80% 以上。

### 統合テスト
- **方法**: RSS/Gemini/SMTP をモック化し、パイプライン結線を検証。
- **対象**: 「取得→差分→要約→HTML→送信」の一連、及び「新着0本で未送信」。

### E2Eテスト
- **ツール**: `workflow_dispatch` による手動実行。
- **シナリオ**: 実チャンネルで実行し、受信メール体裁と seen.json 書き戻し（二重通知なし）を確認。

## 技術的制約

### 環境要件
- **実行 OS**: GitHub Actions `ubuntu-latest`（開発は macOS/Linux）。
- **必要な外部依存**: GitHub リポジトリ（Actions 実行・状態書き戻し）、Gemini API キー、Gmail アカウント + アプリパスワード。
- **ネットワーク**: YouTube RSS / Gemini API / Gmail SMTP への外向き通信。

### パフォーマンス制約
- Gemini 無料枠のレート/日次上限に依存。上限到達時は当該動画を要約失敗扱い（代替表示）とし、全体は継続。
- Gmail の1日あたり送信上限（個人アカウントの制限）内で運用。1日1通のため通常問題にならない。

### セキュリティ制約
- Secrets をコードに埋め込まない・出力しない。
- 自動コミットは seen.json など非機密ファイルのみを対象とする。

## 依存関係管理

| ライブラリ | 用途 | バージョン管理方針 |
|-----------|------|-------------------|
| httpx | RSS/HTTP 取得 | `requirements.txt` で範囲指定（`~=`） |
| feedparser | RSS パース | 範囲指定 |
| google-genai | Gemini 要約 | **固定**（モデル互換性のためピン留め） |
| pytest / ruff | 開発ツール | 開発用途、範囲指定 |

依存は最小限に保ち、新規追加はユーザー承認のうえで行う。
